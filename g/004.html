Weekly Engineering Update

RMLEB IIP Status Report

Project: Internal Information Portal | Date: Dec 3, 2025

Executive Summary

This week marked a significant milestone in the maturity of the RMLEB IIP platform. The engineering team successfully transitioned the database architecture from raw SQL execution to SQLAlchemy Core, establishing a robust foundation for future scalability and type safety.

Simultaneously, we delivered high-value visual analytics for the Transplant Placement and Donor Recovery modules. Stakeholders can now view real-time Domestic vs. International shipping reach, placement utilization benchmarks, and granular donor authorization statistics. These features directly support the CPR team's hospital engagement strategies and the DRC's operational oversight.

Key Highlights

KPIs & Visualization

  • Real-time Domestic vs. International Shipment Reach.
  • Placement Benchmarking (Utilization Rate vs. Global Average).
  • Total Shipped Tissues KPI (Transplant, Research, Training).

Architecture & Stability

  • Major Migration: PyODBC replaced with SQLAlchemy Core.
  • Improved Connection Pooling & Error Handling.
  • Standardized "Sync" Endpoints for Thread-Safety.

Donor Analytics

  • Composite Charts: Referrals vs. Auth vs. Registered.
  • Logic Refinement: Verified recoveries only.
  • New Monthly Tissue Suitability categorization.

Quality Assurance

  • Fixed date range boundary (off-by-one errors).
  • Corrected schema typos (EyeTissueRecoveredId).
  • Expanded mock data for robust offline testing.

Impact Analysis

Departmental Benefits

CPR Team
Can now present hospital partners with concrete evidence of reach (Domestic/International maps) and performance benchmarks. The new placement percentage metrics allow for data-driven discussions about the efficacy of donations from specific institutions compared to the global average.
DRC Team
Gains immediate visibility into Tissue Suitability trends and Recovery Authorization rates. The dashboard now differentiates between tissues recovered for Transplant vs. Research/Training, enabling better resource allocation and technician feedback loops.
IT / Development
The SQLAlchemy migration significantly reduces technical debt. It abstracts complex SQL generation, reduces the risk of SQL injection, and handles database connections more efficiently, ensuring the platform remains stable as data volume grows.

Technical Deep Dive

Backend Core & Refactoring

  • SQLAlchemy Migration: Replaced raw string-based query builders in `utils/helpers.py` with SQLAlchemy Core expressions in `db/expressions.py`. This touches `donor_service` and `transplant_service` extensively.
  • Schema Centralization: Moved `DateRange` and `DashboardFilters` to `schemas/common.py` to prevent circular imports and promote reuse.
  • Sync Endpoints: Converted read-heavy API endpoints from `async def` to `def`. This allows FastAPI to run blocking DB operations in a separate thread pool, preventing event loop blocking.

Transplant Module

  • KPI - Reach: Implemented logic to calculate Domestic vs. International shipping based on `Organization.CountryCode`.
  • KPI - Utilization: Added logic for Placement Percentage (Placed / Recovered) and Global Benchmarking (ignoring filters).
  • Disposition Aggregation: Consolidated multiple API calls into a single `get_disposition_metrics` endpoint for performance.

Donor Module

  • Composite Logic: Updated `get_monthly_composite` to join `DonorDesignation` and `Recovery`. Now accurately counts Registered vs. Authorized donors only when an actual recovery record exists (`etr.Id IS NOT NULL`).
  • Date Handling: Switched SQL filtering to half-open intervals (`>= start AND < end + 1 day`) to correctly capture records with timestamps on the end date.
