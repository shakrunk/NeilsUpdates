üìã Executive Summary

This reporting period represents a pivotal maturation for the RMLEB IIP, driven by the rollout of live data for the Transplant Placement module and a foundational modernization of the backend architecture.

The CPR Dashboard now delivers real-time logistics tracking, enabling stakeholders to visualize Domestic vs. International reach and "Total Shipped Tissues" KPIs directly from the database. This was coupled with the implementation of a sophisticated benchmarking system, allowing specific institution performance to be compared against global averages.

On the technical front, we successfully migrated the entire data access layer from raw PyODBC string queries to SQLAlchemy Core. This significantly enhances maintainability, security, and connection pooling efficiency. Concurrently, a new global filtering system was deployed, empowering users to slice data by precise date ranges, organization categories, and specific institutions.

‚úàÔ∏è Transplants & Logistics

Shipment & Reach Analytics

The Transplant Placement tab has been upgraded to provide a comprehensive view of tissue movement.

  • Domestic vs. International: Implemented logic to calculate reach based on shipping destination country code (US vs Non-US).
  • Total Shipped KPI: Added endpoints to calculate total tissues shipped for Transplant, Research, or Training.
  • Live Data Integration: Replaced `TRANSPLANTS_CHART_US_MAP` mocks with `TRANSPLANTS_KPI_DOMESTIC` real-time feeds.

Disposition & Placement Metrics

  • Disposition Charting: Visualizes tissue outcomes (Suitable, Placed, Pending, Expired) using new aggregation endpoints.
  • Utilization Rates: Calculates Placement / Recovery ratios with new `PlacementData` schemas.
  • Benchmark Comparison: Backend logic now calculates a "Global Benchmark" (clearing filters) alongside specific institution performance for context.

‚öôÔ∏è Backend Architecture

SQLAlchemy Core Migration

A massive refactor was undertaken to modernize the data access layer.

  • Removed PyODBC Strings: Replaced error-prone string concatenation with Pythonic SQLAlchemy table definitions and expressions.
  • Connection Pooling: Implemented `QueuePool` in `DatabaseManager` for efficient resource usage.
  • Refactored Services: Completely rewrote `donor_service` and `transplant_service` to use the new expression language.

Advanced Filtering Engine

  • DashboardFilters Schema: Centralized Pydantic model handling `date_range`, `categories`, and `institutions`.
  • Common Table Expressions (CTEs): Implemented reusable CTEs for complex filtering (e.g., derived organization categories) to improve query readability and performance.
  • Sync Endpoints: Converted database-heavy endpoints from `async def` to `def` to properly utilize FastAPI's thread pool for blocking I/O operations.

üìä Donor Analytics

Demographic Refinement

  • Granular Age Buckets: Shifted from 20-year to 10-year age intervals (0-9, 10-19, etc.) for better demographic insights.
  • Recovery Status: Charts now explicitly split data into `recovered_count` and `unrecovered_count` based on `EyeTissueRecovered` table joins.

Composite Metrics

  • Authorization & Registration: Implemented business logic to count "Authorized" (NotReg + Auth Source) and "Registered" (Registry Status 'Y') donors.
  • Yearly/Monthly Data: Connected composite charts to live APIs, supporting date range filtering.

üé® Frontend Architecture

Client & State Management

  • API Client Pivot: Refactored `CPRClient` to use domain-specific methods (e.g., `getTransplantDomestic`) rather than generic CRUD.
  • POST-based Fetching: Switched read operations to `POST` to support complex JSON filter payloads in the request body.
  • Global Filter Context: Implemented `FilterProvider` to manage state for Date, Category, and Institution selections across all tabs.
  • Mock Data Indicators: Added `isMocked` prop to UI cards to visually warn users when live data is unavailable.
